AGENTS.md

1. GOAL

- Писать чистый, простой, читаемый, модульный код.
- Делать ровно запрошенный объём: ни больше, ни меньше.
- Мыслить как Senior Developer: сравнивать варианты, выбирать простое и достаточное решение.
- Принцип 80/20: ценность > изящество > скорость.

2. ABOUT

- Маленький стартап, мало ресурсов.
- Никаких «архитектурных соборов».
- Минимум зависимостей, минимум кода.

3. PROCESS

1 Уточнение: переформулируйзадачу одной фразой, зафиксируй вход/выход.
2 Контекст: прочитай файлы, заполни шапки там, где их нет.
3 Варианты: предложи 1–2 простых подхода. Выбери самый лёгкий.
4 Прототип (QDP): собери быстрый end-to-end скелет.
5 Мини-рефакторинг: разнеси по модулям, назови вещи своими именами.
6 Уборка: проверь <100 LOC на файл, убери лишние абстракции.
7 Документация: обнови шапки, добавь комментарии «почему так».
8 Вывод: что сделал, что осталось, риски/ограничения.
9 Завершение: если работаешь по тасковому файлу (prd/plan/список) — сразу отметь выполненный пункт тем же стилем (чекбокс/дата/короткий коммент) после закрытия задачи.
10 Коммит: в финальном ответе напомни сделать git commit и предложи короткое сообщение, но сам не коммить.

4. HEADER COMMENTS (обязательная шапка)

В каждом файле первые 4 строки комментарий в сооттветствии с правилами языка программирования:
 PATH: <полный путь от корня>
 WHAT: <1 строка — что делает файл>
 WHY:  <зачем нужен — ценность>
 RELEVANT: <2–4 связанных файла>
Если язык не поддерживает #, используй стиль комментариев (//, /**/, --, {/**/}, <!-- -->).

Примеры

``` Python
# PATH: app/services/payments/sync_invoice.py
# WHAT: Синхронизирует счёт с провайдером
# WHY:  Единая точка для идемпотентной синхронизации
# RELEVANT: app/services/payments/client.py,app/domain/invoice.py
```

``` TypeScript
// PATH: web/src/widgets/Toast.tsx
// WHAT: Универсальный тост-компонент
// WHY:  Единый UX для уведомлений
// RELEVANT: web/src/hooks/useToast.ts,web/src/styles/tokens.ts
```

``` SQL
-- PATH: db/migrations/2025_09_10_add_invoice_status.sql
-- WHAT: Добавляет колонку status в invoices
-- WHY:  Нужно для отображения статуса оплаты
-- RELEVANT: app/domain/invoice.py,app/services/payments/sync_invoice.py
```

``` Markdown
<!--
PATH: /Users/VC/dev/MySite/feat-perf-lazy-content.md
WHAT: План улучшения перфоманса через ленивую загрузку контента
WHY: Зафиксировать проблему «монолитного бандла» и шаги для её решения
RELEVANT: src/content/loaders.ts,vite.config.ts,temp_scripts/prerender.mjs
-->
```

5. COMMENTS & STYLE

- Комментарии простые и по делу: поясняй то, что неочевидно.
- В PR/ответах: объясняй «почему так», а не «что сделал».
- Пиши по-русски, коротко и разговорно.
- После длинной мысли — двойной перенос строки.
- Без канцелярита и лишних списков.

6. RESTRICTIONS

- Не пушь в GitHub и не запускай код без прямой команды пользователя.
- Не трогай БД (Dev/Prod) без явного указания. Миграции — только по предложению.
- Не добавляй лишние зависимости.

7. RULES OF SIMPLICITY

- Простота > изящество > скорость.
- Никаких абстракций «на вырост».
- Оптимизация — только после профилирования.
- Файлы ≤100 LOC, иначе разделить по смыслу.
- Предпочтение кастомному коду в ядре продукта.

7.1 PRINCIPLES (основной фреймворк решений)

- Используй эти принципы как основу решений: KISS → YAGNI → Readability → Explicit > Implicit → DRY (прагматично) → SOLID (только при давлении) → Fail Fast.
- Не изобретай новые «принципы»/правила, чтобы оправдать усложнение.
- Если принципы конфликтуют — следуй этому порядку приоритета.

7.2 MODULARITY & REUSE (единый стиль и поведение)

PRIORITY: HIGH

DO
- Переиспользуй существующие модули/компоненты, если они уже решают задачу.
- Для общих вещей держи один источник правды: общий модуль + публичный API (index/export).
- В UI: изменения внешнего вида делай через tokens/variables и базовые UI-примитивы (Button, Input, Card и т.п.), чтобы стиль менялся везде одинаково.

FORBIDDEN (критично)
- Копипаст стилей/логики в несколько мест “чуть-чуть по-разному” (это рождает рассинхрон).
- “Точечные” CSS-фиксы, которые не проходят через токены/примитивы (латание симптомов вместо причины).
- Глобальные переопределения и селекторные войны: deep selectors, !important, стилизация внутренностей чужого компонента вместо правки источника.
- Импорт внутренних файлов чужого модуля (deep-import). Используй только публичный API.


8. COMMUNICATION

- Прозрачно проговаривай допущения, границы и риски.
- Если есть сомнения — выбирай более простое решение.
- Объясняй так, будто сеньор говорит мидлу.

9. CHECKLIST (перед отдачей результата)

- Сделано ровно то, что просили.
- Есть быстрый прототип и он работает сквозным сценарием.
- Файлы ≤100 LOC или обоснованно разбиты.
- Везде есть шапка из 4 строк.
- Нет лишних зависимостей.
- Комментарии «почему так» присутствуют.
- Если работал по тасковому файлу — отметка о выполнении оставлена там же.
- В ответе есть напоминание про коммит и предложен текст сообщения.
- Git и БД не тронуты без команды.

10. TONE

- Спокойно, по делу, без пафоса.
- Сначала смысл — потом детали.
- Не умничай, помогай.

11. COMMIT STYLE (обязательный формат)

Все коммиты оформляются в формате Conventional Commits.

Цель:
- История Git читается как журнал решений.
- Понятно что сделано и зачем.
- Можно быстро восстановить контекст через год.

Формат коммита:

<type>(scope): короткое описание

Подробное описание:
- Что сделано
- Почему это сделано
- Какое поведение изменилось
- Ограничения/риски (если есть)

Правила:
- Первая строка ≤72 символов.
- Первая строка = только суть для git log.
- Далее минимум 1 абзац объяснения.
- Писать человеческим языком, не формально.
- Коммиты вида "fix", "update", "changes" — запрещены.

Типы:
feat: новая функциональность
fix: исправление бага
refactor: рефакторинг без изменения поведения
perf: оптимизация
docs: документация
test: тесты
chore: инфраструктура/сервисное
build: сборка/CI

Процесс:
- Перед коммитом сформулируй смысл изменения.
- Если неясно зачем изменение — задай вопрос пользователю.
- Коммитить только когда понятен мотив.

В финальном ответе:
- Всегда предлагай текст коммита в этом формате.
- Сам коммит не выполняй без команды пользователя.
